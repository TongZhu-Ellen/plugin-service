# 设计演进说明

本文记录当前系统的已知局限，以及针对三个核心问题的演进思路：插件隔离、版本降级、热加载。

---

## 一、插件隔离：如何真正防住恶意插件

当前系统的隔离边界停在 Go runtime 层面。`recover` 能兜住 panic，`context` 能放弃等待，但插件调用 `os.Exit()` 或死循环占满 CPU 时，主程序无能为力。根本原因是插件和主程序跑在同一个进程里，共享内存和生命周期。

要真正隔离，必须把插件推到进程外面去。

**v2.0 — 子进程隔离**

每次执行插件时，主进程 fork 一个子进程，在子进程里跑插件逻辑，通过 stdin/stdout 传递 JSON 格式的输入和输出。

真正实现的进程隔离：子进程 `os.Exit()` 只杀死自己，`panic`只炸自己；而主进程可以直接 `kill` 子进程。达到真正意义上的“可拔插”。

局限：每次执行都要 fork + 序列化/反序列化，性能开销明显。



**v3.0 — 沙箱隔离**

把插件跑在 WebAssembly 沙箱或 Docker 容器里。但沙箱需要编译成为Wasm，而Docker响应时间较长。

都不是完美的方案。但是足够安全。

## 二、版本管理与降级

当前 Registry 用 `name@version` 作为唯一 key，同一插件的不同版本可以共存，但没有版本间的关联关系，无法做自动降级。

**改进方向**

在 Registry 的数据结构里，将同名插件的多个版本组织为LinkedList形式，按语义版本号从高到低排列（如 `v2.1 → v2.0 → v1.0`）。

执行时，优先选取最高可用版本。如果该版本执行失败则自动尝试下一个版本，直到某个版本成功或全部失败。

（这个机制不需要拓扑排序。拓扑排序是用来处理插件间依赖关系的（插件 A 必须在插件 B 之后执行），本系统的插件彼此独立，不存在这个问题。）

## 三、热加载

热加载指在主程序运行期间，动态加载新插件或卸载已有插件，无需重启进程。

**Go plugin 包方案的现实问题**

- 编译强耦合：`.so` 必须与主程序使用完全相同的 Go 版本和所有依赖版本，否则加载失败。随着插件数量增加，维护所有插件的编译环境与主程序保持一致，工作量线性上升。
- 加载失败会导致插件系统整体不可用（而非单个插件失效）。
- Windows 不支持。

**本系统的现状**

当前采用编译期引入（`init` 注册），不支持热加载。权衡后认为热加载带来的复杂度和维护成本，在当前场景下不值得。
